# 技術仕様書 (Architecture Design Document)

## テクノロジースタック

### 言語・ランタイム

| 技術 | バージョン | 選定理由 |
|------|-----------|----------|
| Node.js | v22.x (LTS) | GitHub Actions / Astroのビルド実行環境として安定した長期サポート版 |
| TypeScript | 5.x | Astroのデフォルト言語。コンテンツ型の安全な定義とIDEの補完で品質向上 |

### フレームワーク・ライブラリ

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| Astro | 4.x | 静的サイトジェネレーター | コンテンツ中心サイトに最適化・コンテンツコレクションでMarkdown管理・デフォルトでゼロJS |
| Tailwind CSS | 3.x | スタイリング | ユーティリティファーストで高速なUI実装・未使用CSSを自動パージ |
| Shiki | Astro組込 | コードシンタックスハイライト | Astro標準搭載・JSゼロ・多言語対応 |

### インフラ・CI/CD

| 技術 | 用途 | 選定理由 |
|------|------|----------|
| GitHub Pages | 静的サイトホスティング | 無料・HTTPS自動対応・GitHubとの統合が容易 |
| GitHub Actions | CI/CDパイプライン | GitHub Pages へのデプロイを自動化・追加コスト不要 |

### 開発ツール

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| ESLint | 9.x | コード品質チェック | Astro公式プラグインがあり設定が容易 |
| Prettier | 3.x | コードフォーマット | Astro / Tailwindプラグイン対応で統一フォーマットを維持 |
| Lighthouse CI | 最新 | パフォーマンス計測 | PR時にスコアを自動チェックし品質基準を維持 |

## アーキテクチャパターン

### 静的サイトアーキテクチャ (Jamstack)

```
┌─────────────────────────────────────────────┐
│   コンテンツレイヤー                          │
│   Markdown記事 / TypeScriptデータファイル      │
├─────────────────────────────────────────────┤
│   ビルドレイヤー (Astro SSG)                  │
│   コンテンツ取得 → テンプレート適用 → HTML生成  │
├─────────────────────────────────────────────┤
│   ホスティングレイヤー (GitHub Pages)          │
│   静的ファイル配信 (HTML / CSS / 画像)         │
└─────────────────────────────────────────────┘
```

#### コンテンツレイヤー
- **責務**: スキル・経歴・プロジェクトデータの管理、および Zenn 記事のビルド時取得
- **形式**: TypeScript (構造化データ) + Zenn RSS フィード (記事)
- **更新方法**: TypeScript ファイルを編集して Git に push（構造化データ）／ Zenn に記事を公開してビルドを実行（記事）

#### ビルドレイヤー
- **責務**: コンテンツを静的HTMLに変換
- **実行タイミング**: GitHub Actionsのワークフロー (`main` ブランチへのpush時)
- **出力**: `dist/` ディレクトリに静的ファイルを生成

#### ホスティングレイヤー
- **責務**: 静的ファイルのグローバル配信
- **プラットフォーム**: GitHub Pages
- **更新**: GitHub Actionsがビルド後に自動デプロイ

### CI/CDパイプライン

```
mainブランチへのpush
        │
        ▼
  GitHub Actions起動
        │
        ├── npm ci (依存インストール)
        ├── npm run check (型チェック・lint)
        ├── npm run build (静的ファイル生成)
        ├── Lighthouse CI (パフォーマンス計測)
        └── GitHub Pagesへデプロイ
```

## データ永続化戦略

### コンテンツの管理方式

| データ種別 | 管理方法 | 形式 | 理由 |
|-----------|---------|------|------|
| 技術記事 | Zenn RSS フィード (ビルド時 fetch) | `https://zenn.dev/{username}/feed` | Zennで記事管理・手動管理不要・ビルド時に最新を取得 |
| スキル一覧 | TypeScriptファイル | `src/data/skills.ts` | 型安全・ビルド時にバリデーション |
| 職務経歴 | TypeScriptファイル | `src/data/career.ts` | 型安全・ビルド時にバリデーション |
| 制作物 | TypeScriptファイル | `src/data/projects.ts` | 型安全・ビルド時にバリデーション |
| 画像 | 静的アセット | `public/images/` | Astroの画像最適化で自動WebP変換 |

### バックアップ戦略

- **リポジトリ管理**: GitHubにすべてのコンテンツをバージョン管理
- **復元方法**: `git checkout <commit>` で任意の時点に復元可能
- **データ損失リスク**: Gitのログが失われない限りゼロ

## パフォーマンス要件

### ページ表示速度

| 指標 | 目標値 | 測定方法 |
|------|--------|---------|
| Lighthouse パフォーマンス | 90点以上 | Lighthouse CI (GitHub Actions) |
| Lighthouse アクセシビリティ | 90点以上 | Lighthouse CI (GitHub Actions) |
| Lighthouse SEO | 90点以上 | Lighthouse CI (GitHub Actions) |
| LCP (最大コンテンツ描画) | 2.5秒以内 | Lighthouse (モバイル) |
| CLS (累積レイアウトシフト) | 0.1以下 | Lighthouse |

### ビルド速度

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| `npm run dev` (初回起動) | 10秒以内 | ローカル開発環境 (M1相当以上) |
| `npm run build` | 60秒以内 | GitHub Actions (Ubuntu) |
| ホットリロード (記事編集) | 1秒以内 | ローカル開発環境 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| 生成HTMLサイズ (1ページ) | 100KB以下 | 初期表示速度の確保 |
| 画像 (1枚あたり) | 200KB以下 | WebP変換後の目標値 |
| JavaScript (バンドルサイズ) | 50KB以下 | 原則ゼロJS、インタラクション最小限 |

## セキュリティアーキテクチャ

### データ保護

- **個人情報**: メールアドレスはHTML直書きせず、コンタクトフォームはFormspree等の外部サービスを利用
- **HTTPS**: GitHub Pagesの標準機能で対応 (追加設定不要)
- **機密情報**: APIキーや認証情報はリポジトリに含めず、GitHub Secretsで管理

### 依存関係のセキュリティ

- `npm audit` をCI/CDに組み込み、高リスクの脆弱性が検出された場合はビルドを失敗させる
- Dependabotを有効化して依存パッケージの自動更新PRを受け取る

### 入力検証

- コンテンツはビルド時にAstroのコンテンツコレクションスキーマで型バリデーション
- 不正なフロントマターはビルドエラーとして検出される

## スケーラビリティ設計

### コンテンツ増加への対応

- **想定記事数**: 100記事程度
- **ページネーション**: ブログ一覧は1ページ20記事ずつページネーション
- **ビルド時間**: 記事数が増えてもAstroの増分ビルドにより線形に増加させない

### 機能拡張性

- **フレームワーク非依存のコンテンツ**: MarkdownベースのためSSG変更時もコンテンツは移行可能
- **カスタムドメイン**: GitHub PagesはCNAME設定でカスタムドメイン対応可能

## テスト戦略

### ビルドチェック (CI必須)
- **型チェック**: `astro check` でTypeScriptの型エラーを検出
- **Lint**: ESLintで構文・コーディング規約を検証
- **ビルド成功確認**: `astro build` が正常終了することを確認（Zenn RSS fetch を含む）

### パフォーマンステスト (CI必須)
- **ツール**: Lighthouse CI (`@lhci/cli`)
- **実行タイミング**: `main` ブランチへのデプロイ完了後に実行
  - Lighthouse CI はデプロイ済みURLに対して計測するため、デプロイ後ステップとして実行する
  - ローカル確認は `npm run preview` 起動後に `npx lhci autorun` で計測可能
- **設定ファイル**: `.lighthouserc.json` (リポジトリルートに配置)
- **失敗条件**: パフォーマンス・アクセシビリティ・SEOスコアが90点未満

### リンクチェック
- **ツール**: `astro check` または `lychee` (外部リンクチェック)
- **対象**: 内部リンクの404・外部リンクの有効性

## 技術的制約

### 環境要件

- **ホスティング**: GitHub Pagesの無料プラン (ストレージ1GB・帯域100GB/月)
- **サーバーサイド処理**: 不可 (静的サイトのみ)
- **データベース**: 不可
- **動的API**: 不可 (外部サービスのみ利用可)

### パフォーマンス制約

- 静的サイトのため、リクエスト時の動的処理は行わない
- コンテンツ更新にはビルドとデプロイが必要 (即時反映不可)

### セキュリティ制約

- サーバーサイドの認証・認可機能を持たない
- フォーム送信は外部サービス(Formspree等)に委譲する
