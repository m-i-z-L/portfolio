# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**個人ポートフォリオサイト** - バックエンドエンジニアの技術と経験を発信するパーソナルサイト

### プロダクトコンセプト
- **技術力の可視化**: スキルセット・経歴・プロジェクト実績を整理し、一目で伝わる形で表現する
- **技術ブログ機能**: 日々の学びや知見を記事として発信し、技術コミュニティへ貢献する
- **無料・低コスト運用**: GitHub Pagesを活用し、維持コストゼロで継続的に運用できる

### プロダクトビジョン
バックエンドエンジニアとしての7年間の経験・スキル・思考を、訪問者に的確に伝えるポートフォリオサイトを構築する。
採用担当者やエンジニア仲間が訪問した際に、技術力・人柄・発信力を直感的に理解できるサイトを目指す。
GitHub Pagesを活用して維持コストをゼロに抑え、長期にわたって継続的に更新・運用できる基盤を作る。

### 目的
- 採用・案件獲得に向けて技術スキルと経歴を効果的にアピールする
- 技術記事の発信を通じて知識の整理とブランディングを行う
- エンジニアとしての存在感をオンライン上で確立する

## ターゲットユーザー

### プライマリーペルソナ: 採用担当者 / 技術採用エンジニア
- 候補者のポートフォリオサイトを数分でスキャンし技術力を判断する
- GitHubリポジトリや技術ブログを参考情報として確認する
- 現在の課題: 短時間で候補者の技術レベルと人柄を把握したい
- 期待する解決策: スキル・経歴・実績が一画面でわかる構成

### セカンダリーペルソナ: エンジニア仲間 / 読者
- 技術記事を読みに来るエンジニア
- SNSや技術系コミュニティ経由でアクセスする
- 現在の課題: 信頼できる実践的な技術情報を探している
- 期待する解決策: 実体験に基づく読みやすい技術記事

### サイトオーナー (自分自身)
- バックエンドエンジニア歴7年弱
- ポートフォリオサイトを初めて持つ
- 現在の課題: 自身のスキル・経歴を整理・発信する場がない
- 期待する解決策: 更新・管理が簡単で、継続的に運用できるサイト
- 典型的なワークフロー: 記事執筆 → Markdownで書く → Gitでpush → 自動デプロイ

## 成功指標(KPI)

### プライマリーKPI
- サイト公開: リリース日から1ヶ月以内
- 技術記事公開数: 公開3ヶ月以内に5記事以上
- ページ表示速度: Lighthouse パフォーマンススコア 90点以上

### セカンダリーKPI
- 月間ページビュー: 公開6ヶ月後に月100PV以上
- 記事の更新頻度: 月1記事以上のペースで継続
- GitHub Pagesでの安定稼働率: 99%以上

## 機能要件

### コア機能(MVP)

#### トップページ / ヒーローセクション

**ユーザーストーリー**:
採用担当者として、候補者の第一印象を素早く掴むために、名前・職種・キャッチコピーがファーストビューで確認できる画面が欲しい

**受け入れ条件**:
- [ ] 名前・職種(バックエンドエンジニア)・簡単な自己紹介文が表示される
- [ ] スクロールなしで主要な情報が把握できる(ファーストビュー内に収まる)
- [ ] SNSリンク(GitHub等)へのリンクが表示される

**優先度**: P0(必須)

---

#### スキル・技術スタックセクション

**ユーザーストーリー**:
採用担当者として、候補者の技術スタックを素早く確認するために、使用言語・フレームワーク・ツール類が視覚的に整理されている画面が欲しい

**受け入れ条件**:
- [ ] 使用言語・フレームワーク・インフラ・ツール等がカテゴリ別に表示される
- [ ] 習熟度や経験年数が視覚的に表現されている(バー・タグ等)
- [ ] 一覧を3秒以内に把握できるレイアウトになっている

**優先度**: P0(必須)

---

#### 職務経歴 / 経歴セクション

**ユーザーストーリー**:
採用担当者として、候補者のキャリアパスを理解するために、職歴・担当業務・実績がタイムライン形式で確認できる画面が欲しい

**受け入れ条件**:
- [ ] 企業名・期間・職種・担当業務が時系列で表示される
- [ ] 各職歴に主要な成果・使用技術が記載されている
- [ ] 7年間のキャリアが一画面でスキャンできる

**優先度**: P0(必須)

---

#### 技術ブログ / 記事一覧

**ユーザーストーリー**:
エンジニア読者として、技術的な知見を参照するために、記事の一覧と個別記事ページが欲しい

**受け入れ条件**:
- [ ] 記事一覧がタイトル・投稿日・タグ付きで表示される
- [ ] 個別記事ページでMarkdownで書いた内容が適切にレンダリングされる
- [ ] コードブロックにシンタックスハイライトが適用されている
- [ ] 記事にタグが付けられ、タグでの絞り込みができる

**優先度**: P0(必須)

---

#### プロジェクト / 制作物セクション

**ユーザーストーリー**:
採用担当者として、候補者の実際のアウトプットを確認するために、制作物・OSSコントリビューション等の実績が見られる画面が欲しい

**受け入れ条件**:
- [ ] プロジェクト名・概要・使用技術・リンク(GitHub等)が表示される
- [ ] 3〜5件のプロジェクトが掲載される
- [ ] 各プロジェクトにスクリーンショットまたはアイコンが表示される

**優先度**: P1(重要)

---

#### お問い合わせ / コンタクトセクション

**ユーザーストーリー**:
採用担当者として、候補者に連絡を取るために、連絡先またはコンタクトフォームが欲しい

**受け入れ条件**:
- [ ] メールアドレスまたはコンタクトフォームが設置されている
- [ ] SNSアカウント(GitHub、X等)へのリンクが表示される

**優先度**: P1(重要)

---

### 将来的な機能(Post-MVP)

#### 検索機能

記事数が増えてきた際に全文検索やタグ検索ができる機能

**優先度**: P2(できれば)

#### ダークモード対応

訪問者の好みやOSの設定に合わせたカラースキーム切り替え

**優先度**: P2(できれば)

#### OGP / SNSシェア最適化

記事がSNSでシェアされた際に適切なプレビュー画像・タイトルが表示される

**優先度**: P2(できれば)

#### RSS フィード

技術記事の更新をRSSで購読できる機能

**優先度**: P2(できれば)

## 非機能要件

### パフォーマンス
- ページ初期表示: 3秒以内(一般的なモバイル回線環境で)
- Lighthouse パフォーマンススコア: 90点以上
- Lighthouse アクセシビリティスコア: 90点以上

### ユーザビリティ
- レスポンシブデザイン: スマートフォン・タブレット・デスクトップで適切に表示される
- 主要コンテンツへのアクセス: トップページから2クリック以内で到達できる
- 読みやすいタイポグラフィ: 本文フォントサイズ16px以上

### 信頼性
- GitHub Pages でのホスティング: サービス障害を除き常時稼働
- 記事データの管理: Gitリポジトリで全履歴を管理し、データ損失ゼロを実現

### 運用性
- 記事更新フロー: Markdownファイルをpushするだけで自動デプロイが完結する
- ローカル開発環境: コマンド1つでプレビューサーバーが起動する
- 依存関係の管理: package.jsonまたは相当のファイルで依存を明示的に管理する

### セキュリティ
- HTTPS: GitHub Pages の標準HTTPS機能を使用
- 個人情報: コンタクトフォームを設置する場合は第三者サービス(Formspree等)を利用し、サーバーサイドの実装を持たない

### スケーラビリティ
- 記事数: 100記事以上になっても表示速度を維持できる構成
- 静的サイトジェネレーター: ビルド時に静的HTMLを生成し、サーバー不要で運用できる

## スコープ外

明示的にスコープ外とする項目:
- 有料ホスティング・独自サーバーの運用
- ユーザー登録・ログイン機能
- コメント機能(記事へのコメント)
- 独自の認証・セッション管理
- データベースを使用するバックエンド実装
